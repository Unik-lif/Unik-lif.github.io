intro

SEV-SNP的起源：SEV -> SEV-ES -> SEV-SNP

原理：

  1. 自动对内存加密，hypervisor只能看到用户加密的数据。(SEV)
  2. 每次hypervisor transition时，对于VM register state加密，从而让hypervisor难以看到被活跃使用的数据，保护内存机密性。(SEV-ES)
  3. 对内存的完整性保护，以及其余安全措施。(SEV-SNP)

需求：对于云服务提供厂商的机架不是特别信任。

完整性

SEV memory encryption key is generated from a hardware random number generator, stored in dedicated hardware registers.

Identical plaintext at different memory locations will encrypt differently.

The basic principle of SEV-SNP intergrity is that if a VM is able to read a private page of memory, it must always read the value it last wrote.

简而言之，如果先前在X位置写入A，之后去读X位置的信息仅有两种情况，要么读到A，要么没法读。

SEV-SNP中增添了对于完整性的保护，主要依赖于RMP(reverse map table)来实现，可以仅仅让memory page的拥有者才能写入该page。

其中设计了一些简单的机制，以实现一一映射。

Reverse Map Table

Reverse Map Table: tracks the owner for each page of memory. Store the physical address with its owner info.(System Physical Address)

info:

  1. -> the page is a guest-owned page
  2. -> assigned to this specific guest
  3. -> mapped at this specific GPA.

As for the mappings in AMD, please refer to the 15.25 of AMD64 Manual Volume 2.(Very similar to Intel’s method)

并不是所有的行为均会通过该项目检查，比如hypervisor read。

Page Validation

This feature is introduced for intergerity, to ensure one-to-one mappings of system pages and virtual pages.

Inside each RMP entry is a Validated bit. The key to our demands is: the guest VM should never validate memory corresponding to the same GPA more than once.

简要分析：

对于nested page tables来说，他能做到让每一个GPA仅映射到一个SPA上，但无法做到该SPA被唯一地由该GPA所一一映射着。为了达到单射的目的，我们需要让一个SPA page仅对应一个GPA page，为此设计了图中的流程。

RMP上存的是系统物理页信息，那么答案就很简单。我对每一个SPA page来说，仅通过RMP回溯找到一个下家GPA page之后，经由RMP更新后的用户名对应的用户PVALIDATE操作后，就马上将其置为Valid。而且仅有一次机会Valid。

这就确保一个SPA Page仅能找到一个GPA Page。那么单射就满足了。双射的满足对于完整性来说其意义不言而喻。
